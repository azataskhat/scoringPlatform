# Обоснование скоринговой модели оценки качества OSINT-данных

## Формулировка темы дипломной работы

**«Разработка платформы мониторинга безопасности IoT-устройств на основе OSINT-источников с количественной оценкой качества данных»**

---

## 1. Почему выбрана именно такая скоринговая модель

### 1.1 Теоретическая основа

Скоринговая модель построена на основе фреймворка оценки качества разведывательных данных **NATO STANAG 2022** (Admiralty Code) и адаптированных критериев качества данных из стандарта **ISO/IEC 25012 (Data Quality Model)**.

Модель оценивает качество данных от каждого OSINT-источника по **4 измерениям (dimensions)**, каждое из которых декомпозировано на **подметрики**:

| Измерение | Подметрики | Что оценивает |
|-----------|-----------|---------------|
| **Reliability (R)** — Надёжность | r1, r2, r3 | Можно ли доверять данным источника? |
| **Timeliness (T)** — Своевременность | t1, t2 | Насколько актуальны данные? |
| **Completeness (C)** — Полнота | c1, c2, c3 | Насколько полны и информативны данные? |
| **Accessibility (A)** — Доступность | a1, a2 | Насколько стабилен и быстр источник? |

### 1.2 Формула итогового скоринга

```
S = wR·R + wT·T + wC·C + wA·A

R = (r1 + r2 + r3) / 3     — Reliability (надёжность)
T = (t1 + t2) / 2           — Timeliness (своевременность)
C = (c1 + c2 + c3) / 3      — Completeness (полнота)
A = (a1 + a2) / 2           — Accessibility (доступность)
```

Все подметрики нормализованы в диапазон **[0.0, 1.0]**, итоговый скоринг S также находится в [0.0, 1.0].

### 1.3 Обоснование выбора весов

| Вес | Значение | Обоснование |
|-----|----------|-------------|
| **wR = 0.35** | Надёжность | Наивысший приоритет: в контексте IoT-безопасности ложные данные опаснее неполных. Ненадёжный источник может привести к ложным срабатываниям или пропуску реальных угроз. |
| **wT = 0.25** | Своевременность | Второй по важности: в сфере кибербезопасности устаревшие данные быстро теряют ценность. Уязвимость, обнаруженная через неделю после появления, значительно менее полезна. |
| **wC = 0.25** | Полнота | Равнозначна своевременности: источник, предоставляющий IP-адрес без порта и типа устройства, имеет ограниченную аналитическую ценность. |
| **wA = 0.15** | Доступность | Наименьший вес: это техническая характеристика API, не влияющая напрямую на качество самих данных. Важна, но вторична. |

**Сумма весов:** 0.35 + 0.25 + 0.25 + 0.15 = **1.0** (нормировка обеспечивает корректность шкалы).

Распределение весов отражает приоритет **«достоверность > актуальность ≥ полнота > доступность»**, что соответствует лучшим практикам оценки качества разведывательных данных (Clark, R.M. "Intelligence Analysis: A Target-Centric Approach", 2019).

### 1.4 Сглаживание методом EMA

Для устранения кратковременных флуктуаций применяется **экспоненциальное скользящее среднее (Exponential Moving Average)**:

```
S_new = β · S_raw + (1 − β) · S_previous
β = 0.3
```

Коэффициент β = 0.3 означает, что 70% итоговой оценки определяется историей, а 30% — текущим замером. Это предотвращает резкие колебания скоринга при одноразовых сбоях API или аномальных выборках данных.

## 2. Детальное описание подметрик

### Reliability (Надёжность)

| Метрика | Формула | Описание |
|---------|---------|----------|
| **r1** — Точность | `valid_records / total_records` | Доля записей с валидным IP-адресом и портом > 0. Проверяет базовую корректность данных. |
| **r2** — Согласованность | `\|IPs_source ∩ IPs_others\| / \|IPs_source ∪ IPs_others\|` | Коэффициент Жаккара: если источник обнаруживает устройства, которые подтверждены другими источниками, это повышает доверие. |
| **r3** — Верифицируемость | `devices_with_CVE / total_devices` | Доля устройств, для которых найдены CVE-записи в NVD. Наличие связи с NVD повышает верифицируемость. |

### Timeliness (Своевременность)

| Метрика | Формула | Описание |
|---------|---------|----------|
| **t1** — Задержка обновления | `exp(−λ · hours_since_discovery)` | Экспоненциальный распад: чем старее данные, тем ниже оценка. λ = 0.05 даёт полупериод ~14 часов. |
| **t2** — Частота сбора | `min(1.0, device_count / 100)` | Нормализованный объём выборки как прокси частоты обновления. |

### Completeness (Полнота)

| Метрика | Формула | Описание |
|---------|---------|----------|
| **c1** — Полнота полей | `filled_fields / (9 × records)` | Средняя заполненность 9 ключевых полей: IP, порт, протокол, тип, производитель, прошивка, город, широта, долгота. |
| **c2** — Покрытие типов | `distinct_types / 6` | Покрытие 6 основных типов IoT-устройств: camera, router, sensor, PLC, gateway, server. |
| **c3** — Глубина | `(cve_depth + raw_depth) / 2` | Наличие расширенных данных: CVE-связей и сырых данных (баннеры, версии ПО). |

### Accessibility (Доступность)

| Метрика | Формула | Описание |
|---------|---------|----------|
| **a1** — Доступность API | `uptime_ratio` | Доля успешных запросов к API за период. |
| **a2** — Скорость ответа | `max(0, 1 − avg_response_time / 10)` | Нормализованное время ответа: 0 сек → 1.0, 10+ сек → 0.0. |

## 3. На каких данных проверяется система

### 3.1 Реальные OSINT-источники

Система собирает данные из **4 реальных OSINT-источников** для города Алматы (Казахстан):

| Источник | Тип данных | Запрос | Интервал |
|----------|-----------|--------|----------|
| **Shodan** | IoT-устройства, сервисы, баннеры, уязвимости | `city:Almaty port:80,443,1883,8883,5683,502` | 60 мин |
| **Censys** | Хосты, сертификаты, сервисы | `location.city: Almaty AND services.port: {80, 443, 1883}` | 120 мин |
| **GreyNoise** | Интернет-шум, классификация трафика | `city:Almaty` | 90 мин |
| **NVD (NIST)** | CVE-записи для IoT-протоколов | Ключевые слова: IoT, MQTT, Modbus, camera, router | 360 мин |

### 3.2 Типы обнаруживаемых устройств

- **Камеры видеонаблюдения** (Hikvision, Dahua) — порты 80, 443, RTSP
- **Маршрутизаторы** (MikroTik, TP-Link) — порты 80, 443
- **IoT-сенсоры** — протоколы MQTT (1883, 8883), CoAP (5683)
- **Промышленные контроллеры (PLC)** — протокол Modbus (502)
- **Шлюзы и серверы** — различные сервисы

### 3.3 Верификация данных

Данные проверяются путём **кросс-валидации между источниками**: устройства, обнаруженные одновременно Shodan и Censys по одному IP-адресу, получают повышенный показатель надёжности (r2 — коэффициент Жаккара). CVE-записи из NVD связываются с обнаруженными устройствами для оценки верифицируемости (r3).

## 4. Метрики и результаты, подтверждающие корректность системы

### 4.1 Ожидаемые диапазоны скоринга

| Источник | Ожидаемый S | Обоснование |
|----------|------------|-------------|
| **Shodan** | 0.65–0.85 | Высокая полнота (баннеры, порты, CVE), хорошая скорость API, широкое покрытие. |
| **Censys** | 0.60–0.80 | Хорошая полнота (сертификаты, сервисы), но менее детальные баннеры. |
| **GreyNoise** | 0.45–0.65 | Узкая специализация (классификация шума), не все поля заполнены, нет CVE. |
| **NVD** | 0.30–0.50 | Нет реальных IP-устройств (только CVE-базы), низкая полнота полей устройств. |

### 4.2 Критерии корректности

1. **Ранжирование источников**: источники с более полными данными (Shodan, Censys) стабильно получают более высокий скоринг, чем узкоспециализированные (GreyNoise, NVD). Это подтверждает адекватность весов.

2. **Стабильность оценок**: EMA-сглаживание обеспечивает дисперсию ≤ 0.05 между последовательными замерами одного источника при стабильной работе.

3. **Чувствительность к деградации**: при отключении одного из полей (например, geo-координаты) скоринг снижается пропорционально (падение c1 на ~11%, что транслируется в снижение S на ~2.8%).

4. **Сходимость**: после 3–5 циклов сбора данных скоринг стабилизируется вблизи характерного для источника значения.

5. **Монотонность**: более полные и точные данные всегда дают более высокий скоринг (нет инверсий в модели, все метрики положительно ориентированы).

### 4.3 Пример расчёта (Shodan)

```
Параметры: r1=0.95, r2=0.30, r3=0.20, t1=0.82, t2=0.45, c1=0.78, c2=0.50, c3=0.35, a1=0.95, a2=0.88

R = (0.95 + 0.30 + 0.20) / 3 = 0.4833
T = (0.82 + 0.45) / 2         = 0.6350
C = (0.78 + 0.50 + 0.35) / 3  = 0.5433
A = (0.95 + 0.88) / 2         = 0.9150

S = 0.35×0.4833 + 0.25×0.6350 + 0.25×0.5433 + 0.15×0.9150
S = 0.1692 + 0.1588 + 0.1358 + 0.1373
S = 0.6010

С учётом EMA (предыдущий S = 0.58):
S_new = 0.3 × 0.6010 + 0.7 × 0.58 = 0.1803 + 0.4060 = 0.5863
```

### 4.4 Визуализация результатов

Система предоставляет инструменты для анализа:
- **Dashboard** (`/api/dashboard/stats`) — сводные метрики: количество устройств, уязвимостей, средний скоринг
- **SSE-стрим** (`/api/scoring/stream`) — обновления скоринга в реальном времени
- **История скоринга** (`/api/scoring/results`) — временные ряды для анализа динамики
- **Карта устройств** (`/api/devices/map`) — географическое распределение
- **PDF-отчёт** (`/api/reports/generate`) — формирование аналитического отчёта

## 5. Сравнение с альтернативными подходами

| Подход | Преимущества | Недостатки | Почему не выбран |
|--------|-------------|-----------|-----------------|
| Бинарная оценка (да/нет) | Простота | Не даёт количественного сравнения | Невозможно ранжировать источники |
| Machine Learning (автовеса) | Адаптивность | Требует размеченную обучающую выборку | Нет эталонных данных для обучения |
| AHP (метод анализа иерархий) | Формальность | Высокая трудоёмкость парных сравнений | Избыточная сложность для 4 критериев |
| **Линейная взвешенная модель + EMA** | Прозрачность, интерпретируемость, настраиваемость весов | Веса заданы экспертно | **Выбрано** — оптимальный баланс простоты и выразительности |
